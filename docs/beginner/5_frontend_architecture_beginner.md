# 🖥️ 프론트엔드 아키텍처 — 쉽게 읽기 버전

> 💬 **이 문서는 무엇인가요?**
> **프론트엔드(Frontend)**는 고객이 실제로 보는 웹 화면입니다.
> 상품 목록, 예약 폼, 장바구니 등 눈에 보이는 모든 것이 프론트엔드입니다.
>
> 이 프로젝트는 **Vue3 + Vite + TypeScript** 조합으로 만들어집니다.
>
> 💬 각 기술이 하는 역할:
> - **Vue3**: 화면을 구성하는 JavaScript 프레임워크 (화면을 컴포넌트 단위로 조립)
> - **Vite**: 개발 속도를 빠르게 해주는 빌드 도구
> - **TypeScript**: JavaScript에 타입 시스템을 추가해 오류를 코드 작성 시점에 미리 잡아주는 언어

---

## 1. 구조 설계 원칙

- **전역 상태 관리 (Pinia):**
  로그인한 사용자 정보와 장바구니 내용은 여러 페이지에서 공통으로 필요합니다.
  이를 **Pinia**라는 "공용 데이터 저장소"에 보관해 모든 화면이 언제든 꺼내 볼 수 있게 합니다.
  > 마치 회사의 공용 구글 드라이브처럼, 누구나 접근할 수 있는 데이터 보관함입니다.

- **HTTP 클라이언트 모듈화 (Axios):**
  서버와 통신할 때 **Axios**라는 도구를 사용합니다.
  요청/응답을 가로채는 **Interceptor(인터셉터)**를 설정해 에러 발생 시 자동으로 알림 팝업을 띄움으로써, 각 화면 개발자가 에러 처리를 따로 신경쓰지 않아도 됩니다.

- **코드 스플리팅 (필요할 때만 로딩):**
  웹사이트가 처음 열릴 때 모든 페이지 코드를 한번에 내려받으면 느립니다.
  사용자가 해당 페이지로 이동할 때만 그 페이지 코드를 내려받도록 최적화합니다.

## 2. 디렉토리 구조

```text
/frontend
├── index.html              ← 사이트의 기반 HTML. SNS 공유용 OG 태그, SEO 메타 정보를 여기에 설정.
├── vite.config.ts          ← Vite 빌드 도구 설정 (사이트맵 생성, 경로 단축키 설정 등)
├── tailwind.config.js      ← 디자인 토큰(색상, 폰트 등) 설정 파일
│
└── src/                    ← 모든 화면 소스 코드가 들어있는 핵심 폴더
    │
    ├── main.ts             ← 앱의 시작점. Pinia(데이터 저장소)와 Router(페이지 전환)를 앱에 연결.
    │
    ├── router/             ← 페이지 주소(URL)와 화면을 연결하는 라우터 설정
    │   └── index.ts        → '/products' 접속 시 상품 목록 화면을 보여주는 등의 규칙 정의.
    │                         관리자 전용(/admin) 페이지는 로그인 여부를 여기서 확인 후 차단.
    │
    ├── store/              ← Pinia 공용 데이터 저장소 (여러 화면이 공유하는 데이터)
    │                         auth: 로그인 상태, 사용자 정보
    │                         cart: 장바구니 목록, 수량
    │
    ├── services/           ← 서버 API와 통신하는 코드를 모은 폴더
    │   ├── api.ts          → Axios 인스턴스 설정. 모든 요청에 자동으로 JWT 토큰을 붙이고,
    │   │                     모든 응답에서 에러를 감지해 Toast 알림으로 전달하는 인터셉터 포함.
    │   ├── product.service.ts → 상품 목록/상세 API 요청 함수들
    │   └── auth.service.ts    → 로그인/회원가입 API 요청 함수들
    │
    ├── composables/        ← 여러 화면에서 공통으로 쓰는 재사용 로직 묶음
    │   ├── useAuth.ts      → 로그인, 로그아웃, 토큰 갱신 로직을 하나로 묶은 함수
    │   ├── useToast.ts     → 화면 어디서든 알림 팝업을 띄우는 함수
    │   └── usePagination.ts → 목록 페이지 넘김(페이지네이션) 관련 공통 로직
    │
    ├── views/              ← 각 URL에 대응하는 "페이지 단위" 화면 파일들
    │   ├── HomeView.vue    → 메인 랜딩 페이지
    │   ├── products/       → 상품 목록(PLP), 상품 상세(PDP) 페이지
    │   └── admin/          → 관리자 전용 CRM/상품관리 화면 (로그인+권한 필요)
    │
    ├── layouts/            ← 페이지들이 공통으로 쓰는 "틀(Frame)" 컴포넌트
    │   ├── DefaultLayout.vue → GNB(상단 메뉴) + Footer를 포함한 일반 페이지 틀
    │   └── AdminLayout.vue   → 관리자 화면 전용 사이드바 메뉴를 포함한 틀
    │
    ├── components/         ← 재사용 가능한 작은 UI 부품들 (Design System 연동)
    │   ├── common/         → BaseButton, BaseInput 등 공통 원자/분자 컴포넌트
    │   └── domain/         → ProductCard, DynamicReservationForm 등 서비스 특화 컴포넌트
    │
    ├── types/              ← TypeScript 타입 정의 파일
    │   └── api.d.ts        → API 응답/요청 데이터의 구조를 TypeScript로 정의해 오타나
    │                         잘못된 데이터 사용을 코드 작성 단계에서 미리 방지.
    │
    └── assets/             ← CSS 파일, 이미지 등 정적 소스
```

## 3. 역방향 검토에 따른 보완 요소

> 💬 백엔드 API와 연결했을 때 발생할 수 있는 문제를 미리 설계에 반영한 내용입니다.

### 📌 보완 1: 에러 처리의 일원화 (`services/api.ts`)
- **문제:** 100개의 화면 각각에서 에러 팝업 코드를 별도로 작성하면 코드가 중복되고, 한 군데 고치면 100군데를 다 고쳐야 합니다.
- **해결:** `api.ts`의 Interceptor에서 API 응답의 `success: false`를 감지하면, 자동으로 `useToast().showError(메시지)`를 호출하여 알림을 띄웁니다. 각 화면 개발자는 성공했을 때의 코드만 신경 쓰면 됩니다.

### 📌 보완 2: 비회원 상태 유지 (`localStorage` 활용)
- **문제:** 비회원으로 예약을 진행하다가 새로고침하면 입력한 정보가 날아갑니다.
- **해결:** 비회원 고객의 임시 정보(이메일, 이름 등)는 브라우저의 `localStorage`에 저장합니다.
  > `localStorage`는 브라우저를 닫아도 남아있는 로컬 저장 공간입니다. 회원가입을 하면 이 임시 정보를 지웁니다.

### 📌 보완 3: 동적 폼 데이터 전송 규격 (`DynamicReservationForm.vue`)
- **문제:** 화면에서는 항목별로 따로 입력받지만, 백엔드는 `dynamic_data` 하나의 통합 항목으로 받아야 합니다.
- **해결:** 화면 컴포넌트 내부에서는 각 항목을 자유롭게 수집하고, 전송(Submit) 시 백엔드 규격에 맞게 `dynamic_data` 객체로 묶어서(Packing) 보내는 로직을 컴포넌트 안에서 처리합니다. 백엔드와 화면 양쪽 모두 깔끔한 구조를 유지합니다.
