# 📦 데이터베이스 모델 — 쉽게 읽기 버전

> 💬 **이 문서는 무엇인가요?**
> 우리 서비스가 저장하는 모든 "데이터"의 구조를 정의하는 문서입니다.
> 쉽게 말해, 엑셀 파일의 각 시트(테이블)와 그 컬럼(항목)들을 미리 설계해 놓은 청사진입니다.
> 실제 데이터는 AWS의 클라우드 데이터베이스(RDS + PostgreSQL)에 저장됩니다.

---

## 1. 설계 원칙

> 💬 **데이터베이스를 만들기 전에 세운 3가지 약속입니다.**

- **확장성:** 나중에 기능이 늘어나도 구조를 크게 바꾸지 않아도 되도록 설계합니다.
  > (예: 비회원이 나중에 회원으로 전환해도 데이터가 무너지지 않게 처음부터 여유 있게 설계)

- **표준화:** 모든 테이블(엑셀 시트)에 공통으로 들어가는 기본 항목 4개를 정해두었습니다.
  > 이 항목들은 아래 `공통 Base Model` 섹션에서 설명합니다.

- **모듈화:** 사용자·상품·주문·예약처럼 서로 다른 역할의 데이터는 각각의 테이블로 분리합니다.
  > (하나의 거대한 엑셀 시트가 아닌, 역할별로 시트를 나눈다고 생각하세요)

---

## 2. 공통 Base Model (모든 테이블에 공통으로 들어가는 항목)

> 💬 **모든 테이블에는 아래 4개 항목이 기본으로 포함됩니다.**
> 마치 모든 서류에 공통으로 찍히는 도장(문서 번호, 작성 일시 등)과 같습니다.

- `id` — 데이터 고유 번호 (UUID 방식)
  > UUID는 "9f5a3b2c-..." 형태의 중복 없는 64자리 고유 식별자입니다. 일련번호(1, 2, 3...)보다 보안이 강하고 여러 서버로 확장할 때 충돌이 없습니다.

- `created_at` — 데이터가 처음 생성된 날짜/시간
  > "이 주문은 언제 들어왔나?" 를 알기 위한 항목입니다.

- `updated_at` — 데이터가 마지막으로 수정된 날짜/시간
  > "이 예약 정보가 마지막으로 언제 바뀌었나?" 를 추적할 수 있습니다.

- `is_deleted` — 삭제 여부 (true/false)
  > ⚠️ **Soft Delete(소프트 삭제)** 방식: 데이터를 실제로 지우지 않고 `is_deleted = true`로 표시해 숨기는 방식입니다.
  > 나중에 실수로 삭제된 데이터를 복구할 수 있고, 히스토리 추적도 가능합니다.

---

## 3. 핵심 테이블 명세

> 💬 **테이블은 "엑셀 시트", 컬럼은 "열 제목"이라고 생각하세요.**
> 예약 테이블이면 → 예약자 이름 / 예약 날짜 / 상담 유형 등을 열로 가집니다.

---

### 3.1. Users (사용자 테이블)
> 💬 회원가입을 한 고객과 관리자의 정보를 저장하는 테이블입니다.

| 항목 | 설명 |
|---|---|
| `email` | 이메일 주소 (동일한 이메일은 중복 불가) |
| `password_hash` | 비밀번호 (보안을 위해 암호화하여 저장. 소셜 로그인 사용자는 비워둠) |
| `name` | 고객 이름 |
| `phone` | 전화번호 |
| `role` | 권한 구분. `ADMIN`(관리자) 또는 `USER`(일반 고객) |
| `auth_provider` | 가입 방법. `LOCAL`(자체 가입), `KAKAO`, `NAVER`, `APPLE` |

> 💬 `password_hash`가 **Nullable(비어있을 수 있음)**인 이유:
> 카카오나 네이버로 간편가입한 사용자는 별도의 비밀번호를 입력하지 않기 때문입니다.

---

### 3.2. Products (상품 테이블)
> 💬 홈페이지에서 판매되는 모든 상품(가구, 소품)의 정보를 저장합니다.

| 항목 | 설명 |
|---|---|
| `sku` | 상품 고유 코드 (예: SOFA-001). 재고 관리를 위한 식별자. |
| `name` | 상품명 |
| `type` | 상품 종류. `FURNITURE`(가구) 또는 `ACCESSORY`(소품) |
| `price` | 가격 (소품은 가격 표기. 고가 가구는 "가격 문의"이므로 비워둘 수 있음) |
| `stock_quantity` | 재고 수량 |
| `status` | 판매 상태. `ACTIVE`(판매중) / `HIDDEN`(숨김) / `SOLDOUT`(품절) |
| `category_id` | 이 상품이 속한 카테고리의 고유 ID (연결 고리 역할) |
| `thumbnail_url` | 상품 목록에서 보이는 대표 이미지 주소 |

> 💬 `category_id`가 `Foreign Key`(외래 키)라고 표기된 이유:
> 카테고리 정보는 별도의 테이블에 있고, 상품 테이블은 그 카테고리 ID를 참조(연결)합니다.
> 마치 엑셀에서 "카테고리" 시트의 행 번호를 "상품" 시트에 적어두는 것과 같습니다.

---

### 3.3. Product_Images (상품 이미지 테이블)
> 💬 상품 하나에 이미지 여러 장을 붙이기 위한 별도 테이블입니다.
> Products 테이블에 이미지를 직접 여러 개 넣으면 구조가 복잡해지기 때문에 따로 분리합니다.

| 항목 | 설명 |
|---|---|
| `product_id` | 어떤 상품의 이미지인지 연결하는 ID |
| `image_url` | 이미지 파일의 주소(URL) |
| `display_order` | 표시 순서 (1번 이미지가 첫 번째로 보임) |
| `is_primary` | 대표 이미지 여부 (true이면 썸네일로 사용) |

---

### 3.4. Orders (주문 테이블)
> 💬 고객이 소품을 결제했을 때 생성되는 주문 정보를 저장합니다.
> 회원뿐 아니라 **비회원도 주문**할 수 있도록 설계된 것이 핵심입니다.

| 항목 | 설명 |
|---|---|
| `user_id` | 주문한 회원의 ID (비회원은 비워둠) |
| `guest_email` | 비회원일 경우 입력한 이메일 |
| `guest_phone` | 비회원일 경우 입력한 전화번호 |
| `guest_name` | 비회원일 경우 입력한 이름 |
| `total_amount` | 결제 총금액 |
| `status` | 주문 상태. `PENDING`(결제전) → `PAID`(결제완료) → `SHIPPED`(배송중) → `DELIVERED`(수령) / `CANCELLED`(취소) |
| `pg_transaction_id` | 카드사/PG사가 발급한 결제 고유 번호 (환불 등에 사용) |

---

### 3.5. Order_Items (주문 상세 테이블)
> 💬 주문 1건에 상품이 여러 개 들어올 수 있으므로 별도 테이블로 분리합니다.
> 예: 주문 #123 → 쿠션 2개 + 화분 1개 → 각각 별도 행으로 저장

| 항목 | 설명 |
|---|---|
| `order_id` | 어느 주문에 속한 항목인지 연결 |
| `product_id` | 어떤 상품을 주문했는지 연결 |
| `quantity` | 주문 수량 |
| `unit_price` | 주문 당시의 단가 (나중에 상품가격이 바뀌어도 주문 당시 금액 보존) |

---

### 3.6. Reservations (1:1 방문 상담 예약 테이블)
> 💬 고객이 쇼룸 방문 예약을 신청할 때 생성되는 예약 데이터를 저장합니다.
> 가구 문의인지, 인테리어 시공인지에 따라 수집하는 추가 정보가 달라지기 때문에
> 유연한 구조(JSONB)로 설계하는 것이 핵심입니다.

| 항목 | 설명 |
|---|---|
| `user_id` | 예약한 회원의 ID (비회원은 비워둠) |
| `guest_info` | 비회원일 경우 이름/전화/이메일을 자유로운 형식으로 저장 (JSONB) |
| `reservation_type` | 상담 유형. `INTERIOR`(인테리어), `FURNITURE`(가구), `TOTAL`(토탈) |
| `status` | 진행 상태. `RECEIVED`(접수) → `IN_PROGRESS`(상담중) → `QUOTED`(견적발송) → `CONTRACTED`(계약완료) / `NOSHOW`(노쇼) |
| `expected_date` | 방문 희망 날짜/시간 |
| `dynamic_data` | 예약 시 고객이 입력한 추가 정보 (평형대, 예산, 공사 희망일 등. JSONB 형식으로 저장) |

> 💬 **JSONB**가 무엇인가요?
> 일반 데이터베이스 컬럼은 딱 정해진 형식(숫자, 텍스트 등)만 저장됩니다.
> JSONB는 마치 "메모 칸"처럼 구조가 유동적인 데이터를 자유롭게 담을 수 있는 형식입니다.
> 가구 예약에만 있는 "희망 브랜드" 항목이나, 인테리어에만 있는 "도면 파일 링크" 같은
> 상황별로 다른 정보를 하나의 칸에 깔끔하게 담을 수 있어 확장에 유리합니다.

---

### 3.7. Waitlists (재고 입고 알림 신청 테이블)
> 💬 품절된 상품에 "재입고 알림을 받겠다"고 신청한 고객 목록을 저장합니다.

| 항목 | 설명 |
|---|---|
| `product_id` | 알림을 신청한 상품 |
| `user_id` | 신청한 회원의 ID (비회원도 신청 가능하므로 비어있을 수 있음) |
| `contact_method` | 연락받을 이메일 또는 전화번호 |
| `status` | `PENDING`(알림 대기) / `NOTIFIED`(알림 발송 완료) |
