# 🏗️ 백엔드 아키텍처 — 쉽게 읽기 버전

> 💬 **백엔드(Backend)**는 웹사이트의 "뒷편 서버"입니다. 고객 눈에는 보이지 않지만,
> 상품 정보를 DB에서 꺼내오고, 예약을 저장하고, 결제를 처리하는 모든 연산이 이곳에서 일어납니다.
>
> 이 프로젝트는 **AWS Chalice** 프레임워크 + **서버리스(Serverless)** 방식으로 동작합니다.
>
> 💬 **서버리스란?**
> 24시간 켜놓은 물리 서버 없이, AWS가 요청이 들어올 때만 실행해주는 방식입니다.
> 사용한 만큼만 비용을 내기 때문에 초기 스타트업에 경제적입니다.

---

## 1. 구조 설계 원칙

- **서버리스 최적화:** 서버가 처음 켜질 때(Cold Start)의 대기시간을 줄이기 위해 시작 시 불필요한 로딩을 최소화합니다.

- **역할별 코드 분리 (의존성 분리):**
  요리에 비유하면, 웨이터(라우터)는 주문을 받고, 주방장(서비스)이 요리를 만들고, 냉장고 담당(CRUD)이 재료를 꺼냅니다. 역할을 나눠야 하나만 수정해도 다른 곳에 영향을 최소화할 수 있습니다.

  | 역할 | 폴더 | 하는 일 |
  |---|---|---|
  | 라우터(Controller) | `api/` | URL 요청을 받아 서비스로 전달 |
  | 서비스(Service) | `services/` | 비즈니스 로직 처리 (결제 검증, 알림 발송 등) |
  | CRUD | `crud/` | 데이터베이스에 데이터를 넣고 꺼내는 쿼리 |

- **공통 처리 로직 중앙화:** 에러 응답 방식과 로깅을 한 곳에서만 관리해 일관성을 보장합니다.

## 2. 디렉토리 구조

```text
/backend
├── app.py                    ← 서버의 "현관문". 가장 먼저 실행되며 각 기능의 Blueprint를 등록.
│
├── chalicelib/               ← 실제 기능 코드들의 핵심 폴더
│   ├── api/                  ← URL 창구(라우터). 요청을 받아 서비스에 전달만 함.
│   │   ├── auth.py           → 로그인/회원가입 URL 처리
│   │   ├── products.py       → 상품 URL 처리
│   │   ├── reservations.py   → 예약 URL 처리
│   │   └── webhooks.py       → PG사 결제 알림 수신 처리
│   │
│   ├── core/                 ← 전체 시스템이 공통으로 쓰는 엔진 폴더
│   │   ├── config.py         → DB 접속 주소, AWS 비밀키 등 환경변수 관리
│   │   ├── db.py             → 데이터베이스 연결 열고 닫는 로직
│   │   └── exceptions.py     → 오류 발생 시 약속된 API 형식으로 변환하는 처리기
│   │
│   ├── models/               ← DB 테이블을 Python 코드로 표현한 파일들
│   │   ├── base.py           → 모든 테이블 공통 항목 (id, created_at, is_deleted)
│   │   ├── product.py        → 상품 테이블 구조
│   │   └── ...
│   │
│   ├── schemas/              ← API 요청/응답 데이터의 규격표
│   │   ├── requests/         → 프론트에서 오는 데이터 검증 규격
│   │   └── responses/        → 프론트로 보내는 데이터 규격
│   │
│   ├── crud/                 ← DB에 직접 질의하는 코드만 모은 폴더
│   │   ├── crud_product.py   → 상품 조회/저장/수정 쿼리
│   │   └── ...
│   │
│   └── services/             ← 복잡한 비즈니스 로직 전담
│       └── payment_service.py → PG사 결제 검증, 가상계좌 상태 처리 등
│
├── requirements.txt          ← 필요한 외부 라이브러리 목록
└── .chalice/                 ← AWS Chalice 배포 설정 파일
```

## 3. 역방향 검토에 따른 보완 요소

> 💬 프론트엔드와 연결했을 때 발생할 수 있는 문제를 미리 설계에 반영한 내용입니다.

### 📌 보완 1: 에러 응답 형식 강제 (`exceptions.py`)
- **문제:** 예상치 못한 오류가 날 것(raw) 그대로 프론트로 가면, 화면이 응답을 해석하지 못해 하얀 화면이 됩니다.
- **해결:** `exceptions.py`가 모든 오류를 가로채서 약속된 포맷(`{ success: false, error: {...} }`)으로 변환 후 응답합니다.

### 📌 보완 2: 트랜잭션(Transaction) 관리
- **문제:** 주문 생성 시 "재고 감소"와 "주문 저장" 두 작업 중 하나만 실패하면 데이터가 꼬입니다.
- **해결:** 여러 DB 작업을 하나의 묶음으로 처리합니다. 모두 성공해야 반영되고, 하나라도 실패하면 전부 취소(롤백)됩니다.
  > 계좌이체와 동일한 원리입니다: 내 계좌에서 빠졌는데 상대방 계좌에 입금 실패 시 전부 취소.

### 📌 보완 3: 동적 예약 폼 데이터 저장
- **문제:** 인테리어/가구 예약마다 수집 항목이 달라 모든 경우를 컬럼으로 만들면 대부분이 비어있는 낭비가 발생합니다.
- **해결:** `dynamic_data` 컬럼을 JSONB(자유로운 형식) 타입으로 지정하여 어떤 추가 정보든 유연하게 담을 수 있도록 합니다.
