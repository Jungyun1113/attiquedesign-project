# 🔌 API 명세서 — 쉽게 읽기 버전

> 💬 **이 문서는 무엇인가요?**
> **API(Application Programming Interface)**란, 프론트엔드(웹 화면)와 백엔드(서버) 사이의 "대화 규칙"입니다.
>
> 예를 들어 고객이 상품 목록 화면을 열면:
> 1. 화면(프론트엔드)이 서버(백엔드)에 "상품 목록 좀 줘!"라고 **요청**합니다.
> 2. 서버가 데이터베이스에서 정보를 가져와 "여기 있어!"라고 **응답**합니다.
>
> 이 문서는 그 요청/응답의 형식과 주소(URL)를 정의합니다.

---

## 1. 공통 응답 구조 (Standard Response Format)

> 💬 **모든 API의 응답은 항상 같은 형식을 사용합니다.**
> 화면 개발자가 성공/실패를 예측 가능하게 처리할 수 있도록 약속된 규칙입니다.
> 마치 택배가 성공하면 "배송 완료" 문자, 실패하면 "배송 불가" 문자가 오는 것과 같습니다.

### ✅ 성공했을 때 응답

| 항목 | 설명 |
|---|---|
| `success` | `true` — 요청이 성공했다는 신호 |
| `data` | 실제 내용 (상품 목록, 주문 정보 등) |
| `meta` | 부가 정보. 목록이 여러 페이지일 때 "현재 몇 페이지인지, 전체 몇 개인지" 등 |

> 💬 **meta 안에 들어가는 정보 예시:**
> - `page`: 현재 보여주는 페이지 번호
> - `size`: 한 페이지에 몇 개 보여주는지
> - `total_pages`: 전체 페이지 수
> - `total_count`: 전체 데이터 수

### ❌ 실패했을 때 응답

| 항목 | 설명 |
|---|---|
| `success` | `false` — 요청이 실패했다는 신호 |
| `error.code` | 어떤 종류의 오류인지 (예: `UNAUTHORIZED`=로그인 필요, `PRODUCT_NOT_FOUND`=상품 없음) |
| `error.message` | 사용자에게 보여줄 수 있는 친절한 오류 메시지 |

> 💬 `error.code`를 별도로 두는 이유:
> 메시지는 "한국어 → 영어" 같이 언어나 내용이 바뀔 수 있지만,
> `code`는 변하지 않는 기계가 읽는 코드라서 화면 개발자가 오류 종류에 따라 다른 처리를 할 수 있습니다.
> 예: `UNAUTHORIZED`를 받으면 자동으로 로그인 페이지로 이동시키는 처리 등.

## 2. 권한 및 인증

> 💬 **로그인한 사용자인지 아닌지를 어떻게 구분할까요?**

- **JWT (JSON Web Token):** 로그인하면 서버가 고객에게 "인증서"를 발급합니다. 이 인증서를 이후 요청마다 함께 보내면 서버가 "이 사람은 로그인된 사람이구나"라고 알아봅니다.
  > 마치 행사장 입장 팔찌처럼, 팔찌(토큰)를 보여주면 다시 검표하지 않아도 됩니다.

- **Bearer Token:** 인증서(JWT)를 전달하는 방식의 이름입니다. 요청할 때 `Authorization: Bearer [토큰값]` 형태로 담아 보냅니다.

- **비회원 처리:** 비회원도 대부분의 기능을 사용할 수 있습니다. 비회원일 때는 토큰 대신 이름, 이메일, 전화번호 등 guest 정보를 요청에 직접 담아 보냅니다.

## 3. 도메인별 API 명세

> 💬 **API 주소(URL) 읽는 법:**
> - `GET` = 데이터를 **가져올 때** 사용 (조회)
> - `POST` = 새 데이터를 **보낼 때** 사용 (생성)
> - `PATCH` = 기존 데이터를 **일부 수정할 때** 사용
> - `{id}` = 중괄호 안은 실제 데이터 ID로 교체됩니다. 예: `/products/abc-123`

---

### 3.1. 인증 API (`/api/v1/auth`)
> 💬 회원가입, 로그인 관련 기능입니다.

| 메서드 | 주소 | 설명 |
|---|---|---|
| `POST` | `/api/v1/auth/login` | 로그인. 이메일+비밀번호 또는 소셜 토큰을 보내면 AccessToken과 RefreshToken을 발급해줍니다. |
| `POST` | `/api/v1/auth/register` | 회원가입. |

> 💬 **AccessToken과 RefreshToken이란?**
> - `AccessToken`: 실제 인증에 사용하는 짧은 유효기간의 인증서 (보통 1시간)
> - `RefreshToken`: AccessToken이 만료됐을 때 새로 발급받는 데 쓰는 인증서 (보통 2주)
> 이렇게 두 개로 나누면 AccessToken이 탈취되어도 피해를 최소화할 수 있습니다.

---

### 3.2. 상품 API (`/api/v1/products`)
> 💬 상품 목록 조회, 상품 등록/수정 기능입니다.

| 메서드 | 주소 | 권한 | 설명 |
|---|---|---|---|
| `GET` | `/api/v1/products` | 누구나 | 상품 목록 조회 |
| `GET` | `/api/v1/products/{id}` | 누구나 | 특정 상품 상세 조회 |
| `POST` | `/api/v1/products` | 관리자만 | 새 상품 등록 |
| `PATCH` | `/api/v1/products/{id}` | 관리자만 | 상품 정보/상태 수정 |

**목록 조회 시 필터 옵션 (Query Params):**
> 💬 URL 뒤에 `?옵션=값` 형태로 붙입니다. 예: `/api/v1/products?type=FURNITURE&sort=latest`

| 옵션 | 설명 | 예시 |
|---|---|---|
| `page` | 몇 번째 페이지를 볼지 | `page=2` |
| `limit` | 한 페이지에 몇 개 | `limit=12` |
| `type` | 상품 종류 필터 | `type=FURNITURE` |
| `category_id` | 특정 카테고리만 | `category_id=abc-123` |
| `sort` | 정렬 방식 | `sort=price_asc` (가격 낮은순) |

---

### 3.3. 주문 API (`/api/v1/orders`)
> 💬 소품 결제 주문 관련 기능입니다.

| 메서드 | 주소 | 설명 |
|---|---|---|
| `POST` | `/api/v1/orders` | 주문 생성. 결제 전에 먼저 "가주문"을 만들고 PG사 결제창으로 연결됩니다. |
| `GET` | `/api/v1/orders/me` | 내 주문 목록 조회 (회원만). |
| `GET` | `/api/v1/orders/{order_id}` | 특정 주문 상세 조회. 비회원도 인증용 토큰을 통해 조회 가능. |

> 💬 **주문 생성(POST) 시 보내는 정보 예시:**
> - `items`: 구매할 상품 목록과 수량
> - `guest_info` (선택): 비회원일 경우 이름, 이메일, 전화번호

> 💬 **"가주문(임시 주문)"이 필요한 이유:**
> 카드사/PG사(포트원, 토스페이먼츠)와 통신하기 전에 우리 서버에 먼저 주문 ID를 만들어두어야,
> 결제가 성공/실패했을 때 어떤 주문과 연결된 결제인지 정확히 매칭할 수 있습니다.

---

### 3.4. 예약 API (`/api/v1/reservations`)
> 💬 쇼룸 방문 상담 예약 관련 기능입니다.

| 메서드 | 주소 | 권한 | 설명 |
|---|---|---|---|
| `POST` | `/api/v1/reservations` | 누구나 | 예약 신청 접수 |
| `GET` | `/api/v1/reservations` | 관리자만 | 전체 예약 목록 조회 (CRM 칸반 화면용) |
| `PATCH` | `/api/v1/reservations/{id}/status` | 관리자만 | 예약 상태 변경 (예: 상담중→계약완료) |

**예약 신청(POST) 시 보내는 정보:**

| 항목 | 설명 |
|---|---|
| `type` | 상담 유형 (INTERIOR / FURNITURE / TOTAL) |
| `expected_date` | 방문 희망 날짜/시간 |
| `guest_info` | 비회원일 경우 이름/전화/이메일 |
| `dynamic_data` | 추가 입력 정보. 가구 예약이면 희망 브랜드를, 인테리어면 현장 주소/예산 등 |

---

### 3.5. 결제 웹훅 API (`/api/v1/webhooks`)
> 💬 **웹훅(Webhook)**이라 어렵게 들리지만, 쉽게 말하면 카드사/PG사가 우리 서버에 먼저 연락하는 방식입니다.
> "고객이 결제를 완료했어요!" 라고 PG사(포트원, 토스페이먼츠)가 우리 서버로 자동으로 알림을 보냅니다.

| 메서드 | 주소 | 설명 |
|---|---|---|
| `POST` | `/api/v1/webhooks/payments` | PG사에서 결제 성공/실패/가상계좌 입금을 통보받는 수신 전용 API |

> 💬 **가상계좌 입금 처리가 왜 별도로 필요한가요?**
> 가상계좌는 고객이 ATM에서 현금을 입금하는 방식이라, 실시간이 아닌 약간의 시간이 걸립니다.
> PG사가 입금이 확인되면 이 웹훅 URL에 신호를 보내고, 우리 서버가 주문 상태를 `PAID`로 변경합니다.
